RHIT RM Code Style Guide

Our goal with this style guide is to allow whatever code is written to be easily and fully understood by any future reader. Consistent commenting, formatting, and naming allow for the code to be better understood. When writing code it’s less important to focus specifically on code readability, and more on writing code that works and comments which explain it. Since this style guide mostly focuses on future readability and less formatting, it’s suggested that you use a linter on your code before submitting just to beautify, I recommend C Lang Tidy (http://clang.llvm.org/extra/clang-tidy/)

Multi-line comments for each function

each function has a comment block above it with the same general information

’Line’ is a general term, comments should not exceed 150 characters long per line, so if a comment reaches that continue it on a newline

Line 1

What the function does [what goes in, what goes out], if the function name is self explanatory then the first line should be “functionName()”, our goal is to have almost all of our functions like this, but some it will not be possible for

Line 2

What data is used, not just the data types that are inputs, where specifically you would get that data from (sensor, value you create, passed by other function, ect)

Line 3

How the function works [how does it do your answer to line 1], this should be to a pretty high level of specificity, if someone were to read this description of your code, they could recreate it very closely

Line 4

Design philosophy, this is a very short statement of why you wrote it a certain way, this can be something like “this was the most straight forward”, “this method avoided using nested loops”, or “dji motors require that we do __”, just something so that a future reader would understand why you chose to do something, and would know why anything weird was happening

Naming Conventions

Variables follow lower_upper_upper

Functions follow lowerUpperUpper

Limit to <15 charcters

Contents should be what the data is followed by an identifier

Ex. making a list of all the motors

  • motors = [] would not be acceptable
  • motor_List = [] would be good

Misc. Guidelines

use 'i' as standard incrementer

braces should start on the line of declaration

Ex. int main(){

}

split blocks of code with newlines, each chunk of code must receive a small comment explaining its function except for variable declaration which can be done if there is ambiguity in variable assignment
